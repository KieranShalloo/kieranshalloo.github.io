On screen Keyboard

Brief

Create an on-screen search capable of replacing the remote control as an input device, ensuring that it would work across all YouView devices and products

Client

BT TV

Project Date

2019

Role

Lead Designer

The problem to solve

Goals

As shareholders were starting to invest more in the YouView product, we aligned product roadmaps to create a unified approach across the main investors.

With one eye on the future road map, we were instructed to investigate ways for a user to interact with the software without relying on the remote control.

The first phase for this was an on-screen keyboard, allowing the shareholders to have more freedom with conceptual work of future remote controls for new versions of their hardware

Secondary goals

Although the search was the main function of the onscreen keyboards, we also had to keep an eye on other areas of the interface where a keyboard would be used instead of physical buttons.

One main area would be entering passwords, especially for internet connection set-up. We would need to include an easy way for users to enter special character s which would not be common in normal search queries

If we didn’t move ahead with this project, we were seriously limiting our shareholders with their creative process. We knew there was a desire to remove letters from the physical remote control, and our insights told us that it would lead to confusion and irritation from our userbase.

We also opened up the ability for another way the user could interact with the box, in turn allowing new and improved features such as voice recognition input to be brought into the project.

I was the lead designer on this project, working alongside a wider design team as well as in house development and a mixture of external and internal QA testing team. This allowed us to explore and experiment with prototypes built directly into the set-top box using HTML5 technology, for fast iterations and tweaks.

What?

Competitor Research

As part of the research into this project, the UX&Design team came together to look at competitors and the different styles they had used for their own on-screen keyboard. We made a conscious effort to not limit ourselves to only TV interfaces and had a wide range of examples from across the globe.

These examples were then critically reviewed for their pros and cons, approaching each example with a set of personas to try to get the widest thought process and practicality.

These Pros/Cons were then grouped into parent features and then ranked in order of importance by the team. Together we were able to then come up with a rough concept that we could present back to stakeholders which we were all aligned on.

Gathering Data

We approached this project with a keen goal of getting as much user feedback as possible. We launched an on-screen dialogue asking for people who would be interested in joining a user testing session to communicate with.

The approach for testing was to use two different versions of the keyboard that we had decided upon as a team.

Concepts

I designed a few variations and through selections and narrowed down the design until we reached a common consensus for the design. We had to look at many different flows when it come to the keyboard.

At certain instances such as WiFi password entry, we had to work out how to easily implement the symbol section of the keyboard. As a user may have to flick between uppercase, lowercase, number and symbols we explored how to show all these characters at the same time.

You can see here the variations we took further than sketchbooks and made into visual designs

Alongside the variations, i also crafted variations of the button press animation would be on the onscreen keyboar. This video shows 4 different variations of how i envisioned the interaction could be implemented, with the use of CSS animation.

AB Testing

I wanted to ensure any results we hypothesis would come into fruition so we needed to get the designs in front of as many live users as possible.

Using a third-party agency we recruited several people to be exposed to two different versions of the design. These people would be a range of people from across the TV spectrum. As well as key demographic areas, we also wanted a split between Retail, BT and TalkTalk subscribers, with a smaller number of users who currently used either Sky or Virgin.



This was accomplished with the use of the Framer.JS prototyping tool, which in turn was set up to two different TVs in a living room style enviroment. By utalising a FLIRC USB we could use a remote control to actually control the Framer prototype, further adding weight to our testing in a real world scenario

We would run a participant through a series of tasks, each time noting where and how they navigated the search section they would have been introduced to earlier in the session.



“Imagine you want to watch EastEnders. Could you show me how you would find the latest episode.

Please talk me through your thoughts and decisions”



Once the user had completed these set tasks with the live software, we would then take them through one of the two new versions, alternating with each user and run the same tasks and finally with the second new version. This eliminated any bias on first introduction of a keyboard and it being an overwhelming change in the current interaction pattern.

These sessions were recorded with permission, and then were shared amongst the wider team not undertaking the sessions, to allow them access to real users thoughts and journeys.

We then collated the evidence gained in these user sessions to help guide our design process

Presenting ideas to stakeholders

After gaining the data from our user testing sessions, we had a clear understanding of which way the general consensus fell in regards to the designs. The final design would have to take inspiration from both designs, but with a leaning towards a three-line narrow concept. We were able to adjust the design based on the feedback and then presented our findings back to the relevant stakeholders.

Luckily there was a great deal of trust in the team that we had produced a viable solution to the problem briefed and we got the green light to implement the work into our upcoming sprints.

Implementation

Development

I worked very closely with the development team to ensure that we could get as accurate implementation onto the set-top box as possible. This was one of the main reason we shared our user feedback so widely. To ensure what we designed was technically feasible with the limitations of low powered devices.

We decided that some animations be turned off for these lower powered devices via feature switch on the backend, allowing the higher quality boxes to still shine.

Design Solution

The final designs were implemented into the youview set-top box, with the highest quality CSS animations implemented on the higher-powered BT 4K box.

I also maintained the ability for users with sight problems to have an accessible version of the onscreen keyboard following the styles and conventions of the high contrast mode previously released to the public

Shareholder Implementation

The BT and TalkTalk shareholders took the design in a slight different direction with the end result. Their own design affected shape and colour of the branding but not any of the journeys or interaction patterns
